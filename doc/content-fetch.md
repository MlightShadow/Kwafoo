## 方案一：基于文本密度的启发式算法

### 技术栈
| 层级 | 技术选型 | 用途 |
|------|---------|------|
| 解析引擎 | `lxml` / `html5lib` | HTML解析与DOM树构建 |
| 文本处理 | 正则表达式 + `unicodedata` | 清洗、统计文本特征 |
| 辅助过滤 | 黑名单规则库 | 预过滤明显噪声区块 |

### 实现流程

**第一步：预处理清洗**
- 移除 `<script>`、`<style>`、`<noscript>`、`<iframe>`、`<svg>` 等标签及其内容
- 移除 HTML 注释 `<!-- -->`
- 移除 `display:none` / `visibility:hidden` 的节点（通过 style 属性判断）
- 将连续空白字符归一化为单个空格

**第二步：构建文本密度模型**
- 遍历 DOM 树所有容器节点（div, section, article, main, td 等块级元素）
- 对每个节点计算：
  - **纯文本长度**：去除标签后的字符数（含标点）
  - **有效文本长度**：仅统计中文字符 + 英文单词（过滤符号）
  - **标签密度**：子标签数量 / 文本长度（值越高越可能是导航/排版）
  - **行内标签占比**：`<a>`、`<span>`、`<strong>` 等行内元素占比（过高则可能是工具栏）
  - **标点密度**：标点符号数 / 总字符数（正文通常 5%-15%，列表/导航极低）

**第三步：候选节点筛选**
- 设定阈值：最小文本长度（如 200 字符）、最大标签密度（如 0.5）
- 过滤黑名单特征：
  - 类名/id 含 `nav`、`menu`、`sidebar`、`footer`、`comment`、`ad-`、`share`
  - 链接密度 > 40%（正文区通常 < 10%）
  - 文本平均长度 < 10 字符（碎片化内容）
- 保留满足条件的节点作为正文候选

**第四步：最优节点选择**
- 对候选节点按 **有效文本长度 × 标点密度** 综合评分
- 处理嵌套问题：若子节点已入选，父节点得分衰减（避免选中外层 wrapper）
- 选择评分最高的连续区块（处理分页/多段落情况：合并相邻高分节点）

**第五步：正文提取与后处理**
- 提取选中节点内的文本，保留 `<p>`、`<h1-h6>`、`<br>` 等语义标签用于分段
- 移除残留的广告标记（如 "推荐阅读"、"相关文章" 等关键词后的内容）
- 输出纯文本或保留格式的 Markdown

### 关键优化点
- **增量迭代**：对高分节点向上回溯 2-3 层父节点，检查是否有更大范围的合理容器
- **反作弊**：检测 display 属性动态修改的节点（部分网站用此隐藏正文）
- **多编码兼容**：处理 GBK/UTF-8 混排，识别乱码并尝试修复

## 方案二：基于DOM路径的评分系统

### 技术栈
| 层级 | 技术选型 | 用途 |
|------|---------|------|
| 解析引擎 | `BeautifulSoup` / `lxml` | DOM 操作与选择器支持 |
| 规则引擎 | YAML/JSON 配置 + 正则匹配 | 维护站点特征库 |
| 评分计算 | 加权评分算法 | 多维度融合决策 |
| 缓存层 | LRU Cache | 相同站点复用学习结果 |

### 实现流程

**第一步：构建特征指纹库**
- 预定义标签权重表：
  ```
  article: +50, main: +40, [role=main]: +35
  div.content: +30, div.post: +25, div.entry: +20
  div.sidebar: -40, nav: -50, footer: -60
  ```
- 维护站点特定规则（可选）：针对 top 100 站点存储已知正文 XPath

**第二步：节点特征向量化**
对每个候选节点提取 6 维特征向量：

| 维度 | 计算方式 | 权重 |
|------|---------|------|
| **语义分** | 标签名 + 类名/ID 匹配关键词库 | w1=0.25 |
| **文本量分** | log(纯文本长度) × 标准化系数 | w2=0.20 |
| **结构分** | 兄弟节点中排名（越大越可能是主内容区） | w3=0.15 |
| **密度分** | 1 - (链接数×10 / 文本长度) | w4=0.20 |
| **连续分** | 子节点中 `<p>` 标签的连续出现次数 | w5=0.15 |
| **位置分** | 在页面垂直方向的居中程度（避开顶部导航/底部） | w6=0.05 |

**第三步：分层评分计算**
- **第一层：粗筛**（O(n) 复杂度）
  - 排除明显噪声：footer、header、nav、aside 标签直接过滤
  - 排除文本量 < 100 字符的叶子节点
- **第二层：精排**（仅对粗筛结果）
  - 计算完整 6 维特征向量
  - 加权求和：Score = Σ(wi × feature_i)
  - 对同层级兄弟节点进行归一化（Softmax 或 Rank）

**第四步：上下文验证**
- **父子校验**：若选中节点的父节点得分 > 80% 且文本量更大，上提至父节点
- **兄弟合并**：检查相邻节点是否同属正文（相似类名、连续段落特征），合并输出
- **边界检测**：识别正文结束标记（如 "相关阅读"、"评论区" 等关键词，或链接密度突增）

**第五步：自适应学习（可选增强）**
- 记录人工校正结果，更新该站点的特征权重
- 使用简单贝叶斯或线性回归，根据历史数据微调 w1-w6 的权重配比

### 关键优化点
- **反框架识别**：检测 `div style="overflow:hidden"` 等隐藏溢出内容，强制展开计算
- **动态内容补偿**：对 Vue/React 渲染的页面，先执行基础 JavaScript 触发数据加载（轻量级，非完整浏览器）
- **多候选融合**：当 top-2 得分接近时（差距 < 10%），输出两者的并集并标记置信度

## 两方案核心差异对比

| 对比项 | 方案一：文本密度法 | 方案二：DOM评分法 |
|--------|-------------------|------------------|
| **核心依据** | 统计学特征（文本/标签比例） | 结构语义特征（标签含义+位置） |
| **知识依赖** | 无需先验站点知识 | 需维护标签权重库和站点规则 |
| **抗干扰性** | 对 class/id 随意命名鲁棒 | 对语义化标签依赖较强 |
| **计算重心** | 文本统计计算 | 结构遍历+规则匹配 |
| **调优方式** | 调整阈值参数 | 增删规则条目、调整权重 |
| **最佳场景** | 批量抓取未知站点、博客/新闻通用模板 | 定向抓取已知站点、复杂 CMS 系统 |

两方案可组合使用：先用方案一快速锁定候选区域，再用方案二在候选区内精细化评分，兼顾速度与准确性。
